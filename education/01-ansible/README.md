# Этап 1: Configuration Management (Ansible)

Документация и руководство для первой недели развертывания проекта Minion Bank. Здесь мы изучаем **Ansible** — инструмент управления конфигурациями. 

## 1. Теория: Что нужно выучить

### Что такое Ansible?
Ansible — это система управления конфигурациями (Configuration Management). Она автоматизирует рутинную настройку серверов: установку пакетов, создание пользователей, деплой конфигурационных файлов и запуск сервисов.

**Главные преимущества Ansible:**
1. **Agentless (без агентов):** Ansible работает через обычный SSH (или WinRM для Windows). Тебе не нужно устанавливать никаких специальных "агентов" (демонов) на сервера, которыми ты управляешь. Достаточно, чтобы на управляемых узлах стоял Python (и он стоит по умолчанию почти везде).
2. **Идемпотентность:** Это фундаментальный принцип Configuration Management. Суть в том, что скрипт можно запускать сколько угодно раз, и он не сломает систему, а лишь приведет её в желаемое состояние. Ansible сначала проверяет: "Установлен ли уже пакет Docker?". Если да — он ничего не делает (статус `ok`). Если нет — устанавливает (статус `changed`).
3. **Декларативность:** Ты описываешь *какое состояние* сервера тебе нужно (например: сервис Nginx должен быть запущен), а не *как именно* это сделать шаг за шагом (например: `systemctl start nginx`). Всю логику выполнения берут на себя встроенные модули.

### Основные концепции (Терминология)
- **Control Node (Управляющий узел):** Ноутбук/Сервер (в твоем случае Arch Linux), откуда вызываются команды Ansible.
- **Managed Nodes (Управляемые узлы):** Целевые сервера, которые мы настраиваем.
- **Inventory (Инвентарь):** Файл (формат YAML или INI), в котором списком указаны IP-адреса хостов, их алиасы и разбивка по группам (например, группы `[webservers]`, `[db]`).
- **Модули (Modules):** Код под капотом (в основном на Python), который выполняет конкретную задачу. Например, модуль `ping` - проверяет коннект, `pacman` - управляет установкой в Arch, `user` - создает пользователей, `copy` - копирует файлы на сервер.
- **Tasks (Задачи):** Одно конкретное действие, которое вызывает один модуль. Например: "Используй модуль `pacman`, чтобы установить пакет `docker`".
- **Playbooks (Плейбуки):** Сценарии в формате YAML, которые объединяют Tasks в последовательные логические цепочки.
- **Roles (Роли):** Способ разбить гигантские плейбуки на переиспользуемые компоненты (папки `tasks/`, `vars/`, `templates/`, `handlers/`).
- **Variables / Facts:** Переменные, которые мы сами можем задать, а также "факты" — детальная системная информация (версия ОС, IP, мак-адреса, диски), которую Ansible автоматически собирает (`Gather Facts`) о каждом хосте перед запуском задач.
- **Handlers (Обработчики):** Задачи, которые "слушают" события и срабатывают только если произошли изменения (`changed`). Например: мы изменяем файл настроек (задача выполнилась с изменением), она посылает триггер ("notify:") хэндлеру "Перезапусти Nginx", и хэндлер выполняет этот рестарт в самом конце плейбука.

## 2. Практика: Что конкретно сделать в коде

Поскольку у нас локальный сетап на Arch Linux, давай напишем базовый *Playbook*, который подготовит твою текущую машину к дальнейшим экспериментам (установит Docker, Terraform и т.д.).

### Структура папок
Создай рядом с этим README следующую структуру:
```text
ansible/
├── inventory/
│   └── hosts.yml
├── ansible.cfg
└── setup-node.yml
```

### 1. Конфиг Ansible (`ansible.cfg`)
Это глобальный файл настроек. В нем мы укажем, где лежит инвентарь и выключим проверку SSH ключей, чтобы Ansible нас не доставал при первом коннекте к новым серверам.
```ini
[defaults]
inventory = inventory/hosts.yml
host_key_checking = False
stdout_callback = yaml
```

### 2. Инвентарь (`inventory/hosts.yml`)
Опишем наши группы серверов. У нас пока группа `local`, в которой только наша машина.
```yaml
all:
  children:
    local:
      hosts:
        localhost:
          ansible_connection: local  # Важно: говорим Ansible запускаться локально, без SSH!
          
          # Для примера, если бы мы настраивали удаленный сервер:
          # server1:
          #   ansible_host: 192.168.1.10
          #   ansible_user: root
```

### 3. Сам Playbook (`setup-node.yml`)
Пишем рецепт настройки нашей ноды. В Arch Linux главным менеджером является `pacman`, поэтому мы используем одноименный модуль.

```yaml
---
- name: Setup Minion Bank Infrastructure Node
  hosts: local # Применяем только к группе local из инвентаря
  become: yes  # Выполнять задачи с правами sudo (root)
  
  tasks:
    - name: Обновление кэша pacman и пакетов системы (pacman -Syu)
      pacman:
        update_cache: yes
        upgrade: yes

    - name: Установка базовых инфраструктурных утилит
      pacman:
        name:
          - docker
          - docker-compose
          - git
          - curl
          - terraform
          - helm
          - kubectl
        state: present

    - name: Запуск и добавление в автозагрузку службы Docker (systemctl enable --now docker)
      systemd:
        name: docker
        state: started
        enabled: yes

    - name: Добавление текущего пользователя в группу docker
      # Через lookup мы берем переменную окружения USER, 
      # чтобы não добавить в группу пользователя root (т.к. мы в режиме become)
      user:
        name: "{{ lookup('env', 'USER') }}"
        groups: docker
        append: yes
```

*(Заметка об Arch: пакеты k3d или kind часто ставятся из AUR (например, через `yay`). В Ansible есть модуль для взаимодействия с AUR (kewlfft.aur.aur), но для простоты старта мы можем установить их вручную или простым модулем `command: yay -S kind --noconfirm`)*.

## 3. Как проверить, что это работает

1. **Запуск плейбука:**
   Открой `fish`, перейди в папку `ansible/` и запусти команду:
   ```fish
   ansible-playbook setup-node.yml -K
   ```
   *(Флаг `-K` или `--ask-become-pass` приказывает Ansible интерактивно спросить у тебя пароль от `sudo` перед запуском)*.

2. **Анализ вывода (Логи):**
   - Если задача что-то реально установила или запустила службу — она будет помечена желтым статусом `changed`.
   - Если нужные пакеты уже стояли, а Docker уже работал — она будет зеленой `ok` (идемпотентность в действии). Ничего не сломается.
   
3. **Проверка результата руками:**
   Убедись, что инструменты появились в системе:
   ```fish
   docker --version
   terraform --version
   systemctl status docker
   ```
   *Важно: Если твой юзер был только что добавлен в группу `docker`, права не применятся сразу. Тебе нужно либо перезайти в систему, либо выполнить команду `su - $USER`, чтобы обновить сессию, иначе Docker будет требовать sudo.*
